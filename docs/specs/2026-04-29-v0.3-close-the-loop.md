# v0.3 тАФ Close the Loop

**Status:** approved 2026-04-29

## Goal

Take the audit from diagnostic to actionable. Three features:

1. **Plugin rollup** тАФ when every skill of a plugin is dead, collapse the rows into a single "uninstall the plugin" candidate.
2. **`prune` subcommand** тАФ generate (and optionally execute) the disable commands for dead skills, source-aware.
3. **`suggest` subcommand** тАФ classify HALLUCINATED + MISSING into actionable buckets (tool/skill confusion, external framework, typo, unclassified).

## Approved defaults

| # | Decision | Value |
|---|---|---|
| Q1 | `prune --apply` scope | unlinks for `user`/`agents`; plugin commands print-only |
| Q2 | Plugin rollup threshold | 100% dead |
| Q3 | Suggest placement | separate subcommand (`audit` stays raw) |
| Q4 | Known-tools list location | `src/known_tools.ts`, hardcoded ~15 names |

## Components

### 1. Plugin rollup

Modifies `audit.ts` and `format.ts`.

Add to `AuditReport`:

```ts
interface PluginGroup {
  pluginName: string;
  pluginScope: string;
  totalSkills: number;
  deadSkills: number;
  activeSkills: number;
  invocationCount: number;
  rollupCandidate: boolean;  // all skills dead
}
report.pluginGroups: PluginGroup[];
```

`format.ts` in the DEAD section:
- For each `rollupCandidate=true` plugin тЖТ one row, label `plugin:<name>` with `9/9 skills dead, 0 used`
- All other dead rows (non-rollup plugins, user-level, agents) тЖТ individual

User-level skills always render individually (no rollup).

### 2. `prune` subcommand

New file: `src/prune.ts`.

Source-aware action matrix:

| source | dry-run output | `--apply` |
|---|---|---|
| `user` (symlink under `~/.claude/skills`) | `unlink <path>` | execute |
| `agents` (symlink under `~/.agents/skills`) | `unlink <path>` | execute |
| `plugin` (rollup candidate, all skills dead) | `claude /plugin remove <name>@<scope>` | print only |
| `plugin` (partial тАФ some skills active) | nothing | nothing |
| `project` | nothing тАФ out of scope | nothing |

CLI:

```
skill-graveyard prune                     # dry-run
skill-graveyard prune --apply             # execute unlinks; plugin commands still print
skill-graveyard prune --only user         # filter by source kind
skill-graveyard prune --days N            # window for "dead" classification (default 30)
```

`--apply` rules:
- Each unlink wrapped: `try { unlink } catch (e) { mark failed }`
- Print `тЬУ unlinked <path>` or `тЬЧ <path> тАФ <error>`
- Plugin removal commands always print, never execute (CC-side slash command, fragile to invoke from outside)
- Exit 0 unless any unlink failed

### 3. `suggest` subcommand

New files: `src/suggest.ts`, `src/known_tools.ts`.

`known_tools.ts` тАФ hardcoded set of CC built-in tool names:

```ts
export const KNOWN_TOOLS: ReadonlySet<string> = new Set([
  "Bash", "Read", "Write", "Edit", "Grep", "Glob",
  "Task", "WebFetch", "WebSearch", "TodoWrite",
  "NotebookEdit", "Agent", "MultiEdit", "Skill",
]);
```

`suggest.ts` reads the audit report, iterates HALLUCINATED + MISSING rows, classifies in this order:

1. **tool_confusion** тАФ invokeName (case-insensitive) matches `KNOWN_TOOLS`. `bash`, `Read`, `Write`, etc.
2. **external_framework** тАФ all cwds for this skill are under `<home>/.<X>/` where X тЙа `claude`. Captures paperclip-style. Group label = X.
3. **typo** тАФ Levenshtein distance тЙд2 to any installed skill's `invokeName`. Suggest the closest match.
4. **unclassified** тАФ fallback.

Output per group: count of skills, total invocations, recommendation text.

CLI:

```
skill-graveyard suggest               # full classification
skill-graveyard suggest --json        # machine-readable
skill-graveyard suggest --days N      # window
```

### 4. CLI subcommand routing

`cli.ts` accepts a positional command:

```
skill-graveyard [audit] [options]   # current default
skill-graveyard prune [options]
skill-graveyard suggest [options]
```

When the first positional arg is `audit`/`prune`/`suggest`, dispatch to that handler. When absent, default to `audit`. Each handler parses its own flag subset.

## Out of scope (defer to later)

- Auto-running `claude /plugin remove` тАФ fragile, separate concern
- Project-scoped skill removal тАФ out of scope; respects per-project intent
- Internet-based recommendations (npm registry, GitHub search) тАФ too speculative
- Rollback/undo
- Theme 1 (token cost estimation) and Theme 3 (CC plugin manifest, statusline) тАФ separate releases

## Implementation order

1. Plugin rollup (`audit.ts` + `format.ts`) тАФ smallest, validates data flow
2. CLI subcommand routing (`cli.ts`)
3. `prune` (`prune.ts`)
4. `suggest` + `known_tools.ts`
5. Tests
