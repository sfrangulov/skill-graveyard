# v0.6 — Roadmap

**Status:** drafted 2026-04-29

## Goal

Five features in priority order. The shift from earlier framing: tokenizer accuracy and per-project analytics ship before distribution, so what we publish is a mature tool rather than a v0.2 toy.

## Priority order

| # | Feature | One-line scope |
|---|---|---|
| 1 | honest tokenizer | replace `chars/4` BPE approximation with real tokenizer |
| 2 | per-project breakdown | `projects` subcommand — skill usage per repo |
| 3 | distribution & embed | npm publish + GitHub repo + Claude Code plugin manifest + CI |
| 4 | visual export formats | render reports as html / md / csv |
| 5 | trend / time-series view | weekly sparklines for skill activity |

## Why this order

- **Tokenizer first** — small change, high credibility lift. Cost numbers go from "rough estimate" to defensible. Required before anyone trusts the `cost` subcommand output.
- **Per-project second** — high-leverage analytics for power users. Surfaces skills that are project-scoped vs. globally used, enabling smarter placement decisions (move to `.claude/skills/` per-project vs. keep global).
- **Distribution third** — was originally planned first, but publishing a tool with weak cost numbers and no per-project view sells the wrong story. Ship analytics polish first, then publish.
- **Visual formats fourth** — diversifies output for sharing/CI consumption. Lower urgency than core data quality.
- **Trend view fifth** — nice-to-have for ongoing monitoring. Depends on having enough historical sessions to be meaningful.

## Per-feature notes

These are roadmap-level only. Each feature gets its own dated spec doc when work starts (see `2026-04-29-v0.3-close-the-loop.md` for the per-feature pattern).

### 1. honest tokenizer

- Candidates: `js-tiktoken`, `gpt-tokenizer`. Pick whichever is lighter and accurate enough for Claude tokenization.
- Affects: `src/cost.ts` and any other site using `chars/4`.
- Open questions: which model encoding to use; whether to cache token counts per session-id to avoid re-tokenizing on every audit.

### 2. per-project breakdown

- New subcommand: `skill-graveyard projects`.
- Output: per-project rows showing which skills are used, dead, or hallucinated within that project's session logs.
- Reuses existing audit pipeline; groups by `cwd` (already tracked via `observedCwds`).
- Open questions: default sort (by activity? by dead-skill count?); whether to add a `--project <path>` filter for drilling in.

### 3. distribution & embed

- npm publish under `skill-graveyard` (or scoped name if conflict).
- GitHub repo: README with install/usage, CI on Node 22 (typecheck + tests), license already in place.
- Claude Code plugin: `.claude-plugin/plugin.json` exposing an `/audit-skills` slash command wrapping the CLI.
- Initial published version: v0.6.0 (skip v0.2 framing — v0.6 is the first release).

### 4. visual export formats

- `--format html|md|csv` flag on `audit`, `cost`, `projects` subcommands.
- HTML: standalone single-file with inline CSS, suitable for sharing.
- Markdown: GitHub-flavored tables.
- CSV: raw rows for spreadsheet analysis.
- Open questions: whether output goes to stdout or `--out <file>`; whether to include the visual dashboard (bar charts) in HTML.

### 5. trend / time-series view

- Weekly sparklines per skill — invocation counts bucketed by ISO week.
- New section in `audit` output, or new subcommand `trend`.
- Requires enough session history to be meaningful (probably ≥4 weeks of data).
- Open questions: bucket size (week vs. day); whether to show only skills with activity changes, or all.

## Out of scope (for v0.6)

- Auto-disable / auto-prune logic beyond what v0.3 already does.
- Internet-based recommendations (npm registry search for missing skills).
- Cross-machine session aggregation.
- Real-time / watch mode.

## Implementation order within v0.6

Same as the priority order above. Each feature lands in its own PR with its own spec doc and tests. Distribution (#3) is a hard cut — once published, the contract surface stabilizes, so do it after analytics features are settled.
